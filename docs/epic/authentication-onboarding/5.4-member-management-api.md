# Task 5.4 — Member Management API (Backend)

**Story:** 5.0 — User Invitation & Role Assignment  
**Skill:** backend-lead  
**Priority:** P1 — depends on Tasks 5.2, 5.3  

---

## What to do

Implement the remaining team management endpoints: role change and member removal.

---

## Acceptance Criteria

**Role change**
- [ ] `PATCH /team/:userId/role` — body: `{ role: 'admin' | 'engineer' | 'viewer' }`. Requires auth + Admin role.
- [ ] Validates target `userId` belongs to the same tenant (prevent cross-tenant role change).
- [ ] Updates `tenant_users.role`.
- [ ] Mirrors the role change to Clerk's organisation membership (Clerk SDK `updateOrganizationMembership`).
- [ ] Returns `200 { userId, role }`.

**Member removal**
- [ ] `DELETE /team/:userId` — requires auth + Admin role.
- [ ] Sets `tenant_users.deletedAt = now()` (soft delete).
- [ ] Calls Clerk SDK `deleteOrganizationMembership` to revoke the user's org access immediately.
- [ ] Returns `204 No Content`.

**Testing**
- [ ] Unit test: role change updates `tenant_users.role` and calls Clerk SDK.
- [ ] Unit test: removal sets `deletedAt` and calls Clerk SDK.
- [ ] Unit test: targeting a userId from a different tenant → `404`.

---

## Notes

- The Clerk SDK call is the authoritative revocation mechanism — `deletedAt` in the DB is a local record only.
- After calling `deleteOrganizationMembership`, Clerk automatically invalidates active sessions for that org — no additional session cleanup needed.
