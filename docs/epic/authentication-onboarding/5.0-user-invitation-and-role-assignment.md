# Story 5.0 — User Invitation & Role Assignment

**Epic:** Authentication & Onboarding  
**Priority:** P1 — MVP required (multi-user teams need this before first run)  
**Roles involved:** Admin (acting), Engineer / Viewer (invited)

---

## User Story

As an admin of a new AIDevFlow organisation,  
I want to invite team members by email and assign them a role,  
so that engineers and reviewers can collaborate on workflows and pipeline runs without sharing my credentials.

---

## Acceptance Criteria

**Sending invitations**
- [ ] From Settings → Team, an Admin can invite a user by entering their email address and selecting a role: `Admin`, `Engineer`, or `Viewer`.
- [ ] Multiple invitations can be sent in a single session (add another row before submitting).
- [ ] Invitations are sent via Clerk's built-in organisation invitation flow (email with a magic link).
- [ ] Invitation appears in the team list as "Pending" until accepted.
- [ ] An Admin can resend or revoke a pending invitation.
- [ ] Inviting an email address that already has a pending invitation shows a warning and replaces the pending invite rather than creating a duplicate.

**Accepting an invitation**
- [ ] Invited user receives an email with a magic link scoped to the inviting organisation.
- [ ] Clicking the link brings the user to Clerk sign-up (if no account) or sign-in (if existing account).
- [ ] On acceptance, the user is added to `tenant_users` with the assigned role and redirected to the dashboard.
- [ ] Accepted invitations are removed from the "Pending" list and the user appears in the Active members list with their assigned role.

**Role assignment rules**
- [ ] Roles map to the permission matrix in `docs/requirements.md` Section 2.3:
  - `Admin`: full platform access including plugin config, billing, user management.
  - `Engineer`: workflow authoring, triggering runs, retrying steps. Cannot manage plugins, billing, or users.
  - `Viewer`: read-only (run trace, dashboard, audit log). Cannot trigger or modify anything.
- [ ] An Admin can change the role of any member except their own (self-demotion requires another Admin to exist first).
- [ ] An Admin can remove any member from the organisation. Removed members lose access immediately (Clerk session is invalidated for that org).

**Self-protection rule**
- [ ] The organisation must always have at least one Admin. The last Admin cannot be removed or demoted.

**Permissions enforcement**
- [ ] All role-based restrictions are enforced server-side in the Coordination API — `role` is read from the verified Clerk token, never from the request body.
- [ ] Client-side: action buttons (e.g. "Invite user", "Configure plugins") are hidden or disabled for roles that lack permission, with a plain-language tooltip explaining why.

---

## Technical Notes

- Clerk handles the invitation email delivery and magic-link acceptance. The platform receives a `organizationMembership.created` webhook from Clerk when an invitation is accepted, which triggers creation/update of the `tenant_users` row.
- `tenant_users.role` is the platform's role — it is set from the `role` field sent in the Clerk invitation metadata. Clerk's built-in role system may be used as a mirror, but the platform's own `role` column is authoritative for permission checks.
- Role is included in the Clerk JWT as an organisation membership claim. The Fastify auth plugin extracts and validates it.
- Removing a member triggers a `organizationMembership.deleted` Clerk webhook → platform sets `tenant_users.deleted_at`.

## Permissions matrix reference

| Action | Admin | Engineer | Viewer |
|---|---|---|---|
| Invite / remove users | ✓ | — | — |
| Configure plugins, billing | ✓ | — | — |
| Author workflows (draft) | ✓ | ✓ | — |
| Trigger manual runs | ✓ | ✓ | — |
| Retry failed steps | ✓ | ✓ | — |
| View run trace, audit log | ✓ | ✓ | ✓ |
| Approve / reject human gates | Allowlist-only | Allowlist-only | — |

## Dependencies

- Story 1.0 (tenant row and initial Admin user exist).
- Story 2.0 (sign-in flow — invited users sign in via the magic link).
- Clerk organisation invitation feature enabled in the Clerk project.
- Coordination API: `GET /team`, `POST /team/invite`, `PATCH /team/:userId/role`, `DELETE /team/:userId` endpoints.
- Fastify role-auth middleware (`requireRole('admin')` guard decorator).

---

## Out of Scope for This Story

- SSO/SAML provisioning (auto-creating users from IdP directory — Enterprise tier).
- MFA enforcement for non-Admin roles (post-MVP).
- Audit log entries for invitation events (Platform Infra epic, but noted as a dependency).

---

## Definition of Done

- [ ] Admin can invite a user by email; invitation email is received in staging.
- [ ] Invited user can accept via magic link and access the dashboard with the correct role.
- [ ] Role permissions enforced server-side (API returns 403 for role-unauthorised actions).
- [ ] Admin can change a member's role; cannot demote the last Admin.
- [ ] Admin can remove a member; removed user loses access immediately.
- [ ] Pending invitations visible; resend and revoke work.
- [ ] Unit tests: role guard middleware (403 on insufficient role).
- [ ] Unit tests: last-admin self-demotion prevention.
- [ ] Integration test: invite → accept → role-restricted action round trip.
