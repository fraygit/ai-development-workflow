# Task 2.1 — Next.js Auth Middleware Guard (Frontend)

**Story:** 2.0 — Sign In & Session Management  
**Skill:** frontend-lead  
**Priority:** P0 — depends on Task 1.3  

---

## What to do

Configure `middleware.ts` in the web repo to protect all `/dashboard` and `/onboarding` routes, redirecting unauthenticated requests to `/sign-in`.

---

## Acceptance Criteria

- [ ] `middleware.ts` at the web repo root uses `clerkMiddleware()` from `@clerk/nextjs/server`.
- [ ] All routes under `/dashboard` and `/onboarding` are protected — unauthenticated requests redirect to `/sign-in`.
- [ ] Public routes explicitly allowed without auth:
  - `/` (marketing home)
  - `/sign-in` and `/sign-up` (and Clerk catch-all sub-paths)
  - `/webhooks/clerk` — if proxied through Next.js (otherwise this is a backend concern only)
- [ ] The `matcher` in `middleware.ts` excludes static assets (`/_next/`, `/favicon.ico`, etc.) to avoid unnecessary middleware runs.
- [ ] Accessing `/dashboard` without a session redirects to `/sign-in?redirect_url=/dashboard`.
- [ ] After sign-in, the user is returned to the original URL (Clerk handles this via `redirect_url`).

---

## Notes

- Use `createRouteMatcher` from `@clerk/nextjs/server` to define public vs protected routes.
- Do not use the older `withClerkMiddleware` API — use the current `clerkMiddleware()` pattern.
- The Clerk middleware also attaches the session context used by `auth()` in Server Components.
