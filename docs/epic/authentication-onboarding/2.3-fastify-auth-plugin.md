# Task 2.3 — Fastify Auth Plugin (Backend)

**Story:** 2.0 — Sign In & Session Management  
**Skill:** backend-lead  
**Priority:** P0 — required before any authenticated API route  

---

## What to do

Implement a Fastify plugin that validates the Clerk session JWT on every authenticated route, extracting `tenantId` and `userId` for use by route handlers.

---

## Acceptance Criteria

**JWT verification**
- [ ] All authenticated routes use a `requireAuth` Fastify decorator/hook.
- [ ] The plugin fetches Clerk's JWKS from `https://api.clerk.com/v1/jwks` (or the environment-specific URL) and caches the key set.
- [ ] The `Authorization: Bearer <token>` header is required on all protected routes.
- [ ] Tokens are verified for: valid signature, non-expired `exp`, correct `azp` (Clerk publishable key).
- [ ] On valid token: `request.tenantId` and `request.userId` are set from the JWT claims.
- [ ] On invalid/missing token: returns `401 Unauthorized` with body `{ code: "UNAUTHORIZED", message: "Invalid or missing session token" }`.

**Org context**
- [ ] `tenantId` is resolved from the Clerk `org_id` claim → looked up in `tenants.clerkOrgId`.
- [ ] If no tenant row exists for the org, returns `403 { code: "TENANT_NOT_FOUND" }`.

**Testing**
- [ ] Unit test: valid JWT → `tenantId` and `userId` set correctly.
- [ ] Unit test: expired JWT → `401`.
- [ ] Unit test: missing `Authorization` header → `401`.
- [ ] Unit test: valid JWT but no matching tenant row → `403 TENANT_NOT_FOUND`.

---

## Notes

- Use the `@clerk/backend` npm package for JWT verification — it handles JWKS caching and token validation.
- `tenantId` and `userId` must NEVER be read from the request body — only from the verified JWT.
- The Fastify `decorateRequest` API adds `request.tenantId` / `request.userId` after a successful auth check.
