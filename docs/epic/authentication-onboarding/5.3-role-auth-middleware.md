# Task 5.3 — Role-Based Auth Middleware (Backend)

**Story:** 5.0 — User Invitation & Role Assignment  
**Skill:** backend-lead  
**Priority:** P1 — depends on Task 2.3  

---

## What to do

Implement a `requireRole` Fastify decorator used to guard routes by role, and enforce the last-Admin self-protection rule.

---

## Acceptance Criteria

**requireRole decorator**
- [ ] `requireRole(...roles: Role[])` Fastify hook — checks `request.userId`'s role in `tenant_users`.
- [ ] Returns `403 { code: "INSUFFICIENT_ROLE", message: "You don't have permission to perform this action." }` if the user's role is not in the allowed list.
- [ ] Role is read from the verified Clerk JWT claim or `tenant_users` DB row — never from the request body.
- [ ] Applied to:
  - `POST /team/invite` → requires `admin`
  - `PATCH /team/:userId/role` → requires `admin`
  - `DELETE /team/:userId` → requires `admin`
  - `POST /plugins/*` → requires `admin`
  - `PATCH /onboarding/*` → requires `admin`

**Last-Admin protection**
- [ ] `PATCH /team/:userId/role` — if the target user is the last Admin and new role is not `admin`, returns `409 { code: "LAST_ADMIN", message: "Cannot demote the last admin. Assign another admin first." }`.
- [ ] `DELETE /team/:userId` — if the target user is the last Admin, returns `409 { code: "LAST_ADMIN" }`.
- [ ] The check is: `COUNT(*) WHERE tenantId = ? AND role = 'admin' AND deletedAt IS NULL` = 1 before the operation.

**Self-demotion**
- [ ] An Admin cannot change their own role via `PATCH /team/:userId/role` — returns `403 { code: "SELF_DEMOTION" }`.
- [ ] An Admin CAN remove themselves only if another Admin exists (the last-Admin check covers this).

**Testing**
- [ ] Unit test: Engineer calling `POST /team/invite` → `403 INSUFFICIENT_ROLE`.
- [ ] Unit test: Last Admin demotion → `409 LAST_ADMIN`.
- [ ] Unit test: Admin self-demotion → `403 SELF_DEMOTION`.
- [ ] Unit test: Role change when 2 admins exist → allowed.
