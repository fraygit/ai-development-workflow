# Task 2.5 — 401 / Expired Token Handling (Frontend)

**Story:** 2.0 — Sign In & Session Management  
**Skill:** frontend-lead  
**Priority:** P0 — depends on Tasks 2.1, 2.3  

---

## What to do

Ensure the frontend handles expired or invalid session tokens gracefully — redirecting to sign-in without showing a blank screen or raw error to the user.

---

## Acceptance Criteria

- [ ] The API client (`lib/api-client.ts`) intercepts `401` responses from the Coordination API and triggers a redirect to `/sign-in`.
- [ ] The redirect preserves the current path as `?redirect_url=<current path>` so the user returns after re-authenticating.
- [ ] No raw `401` or JSON error body is ever surfaced to the user — they see only the sign-in page.
- [ ] If the Coordination API is unreachable (network error / `5xx`), a generic error toast ("Something went wrong. Please try again.") is shown — not a redirect to sign-in.
- [ ] Clerk's session auto-refresh (built into `ClerkProvider`) handles token expiry transparently for most cases — the `401` intercept is a safety net for cases where refresh fails.

---

## Notes

- Clerk's SDK handles silent token refresh automatically. The `401` intercept primarily catches cases where the session was revoked (e.g. org removal, sign-out in another tab).
- Use a TanStack Query `QueryClient` global error handler or an Axios/fetch interceptor (whichever the web repo uses) for centralised `401` handling.
- Do not add this logic per-page or per-query — one central intercept only.
