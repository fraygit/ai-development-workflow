# Task 3.3 — Wizard Step 1: Source Control (Frontend)

**Story:** 3.0 — Onboarding Plugin Setup Wizard  
**Skill:** frontend-lead  
**Priority:** P0 — depends on Task 3.2  

---

## What to do

Build the Source Control step form (Step 1 of the wizard), covering GitHub and GitLab options with connection test integration.

---

## Acceptance Criteria

**Form fields**
- [ ] Provider radio selector: "GitHub" / "GitLab".
- [ ] **GitHub selected:** A "Connect GitHub App" button triggers the GitHub App OAuth install flow (opens GitHub in a new tab or redirect). On return, the `installation_id` query param is captured and displayed as "Connected ✓".
- [ ] **GitLab selected:** Two text inputs — "Group Access Token" and "Group namespace" (e.g. `mycompany`). Inputs are masked (`type="password"` for token).
- [ ] "Advanced" toggle — hidden by default; no advanced fields defined for this step yet.

**Connection test**
- [ ] "Test connection" button calls the backend connection test endpoint (Task 3.6).
- [ ] During the test: button shows a loading spinner; form is non-interactive.
- [ ] On success: inline green "Connection successful" message; "Next" button becomes active.
- [ ] On failure: inline red error message with the reason from the API (e.g. "Invalid token — check that scope includes `api` and `read_repository`").
- [ ] The step cannot advance without a passing test.

**Accessibility**
- [ ] All form controls have visible labels.
- [ ] Focus moves to the first field when this step becomes active.
- [ ] Keyboard-navigable: Tab order is logical across all controls.

---

## Notes

- The GitHub App install flow requires the GitHub App to be created and the OAuth callback URL configured — this is a prerequisite for the GitHub path. For local dev, the GitLab path is easier to test without additional OAuth setup.
- Do not send credentials directly to GitHub/GitLab from the browser — the connection test is proxied through the Coordination API (Task 3.6).
