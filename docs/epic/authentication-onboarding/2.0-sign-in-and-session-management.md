# Story 2.0 — Sign In & Session Management

**Epic:** Authentication & Onboarding  
**Priority:** P0 — MVP critical path  
**Roles involved:** Admin, Engineer, Viewer

---

## User Story

As a returning user,  
I want to sign in to my AIDevFlow account and have my session securely maintained,  
so that I can access my organisation's dashboard without re-authenticating on every action.

---

## Acceptance Criteria

- [ ] A "Sign in" link is visible on all public marketing pages.
- [ ] Clerk-hosted sign-in accepts email + password and Google OAuth.
- [ ] On successful sign-in, the user is redirected to the dashboard home (or the last visited dashboard route if a redirect param is present).
- [ ] If the user belongs to multiple Clerk organisations, an org-selection screen is shown before the dashboard (see Story 2.1 — Org Selection, below).
- [ ] The Clerk session token is validated on every authenticated API route in the Coordination API. `tenantId` and `userId` are extracted from the verified token — never from the request body.
- [ ] Session tokens are refreshed transparently by the Clerk SDK without requiring the user to re-authenticate mid-session.
- [ ] Signing out clears the Clerk session cookie and redirects to the public landing page.
- [ ] Expired or invalid tokens return `401 Unauthorized` from the Coordination API; the frontend redirects to sign-in automatically.
- [ ] Admin MFA can be configured via Clerk admin console; it is not enforced in MVP UI (enforcement is an Enterprise-tier feature).

---

## Sub-story 2.1 — Org Selection

As a user who belongs to more than one organisation,  
I want to choose which org to enter after sign-in,  
so that I land in the correct tenant context.

- [ ] If a Clerk session has multiple org memberships, an org picker screen is shown before routing to `/dashboard`.
- [ ] Each org is displayed with its name and the user's role in that org.
- [ ] Selecting an org sets the `activeOrganizationId` in the Clerk session.
- [ ] Switching org from within the dashboard is accessible from the top navigation; switching re-runs the org-picker flow without full sign-out.

---

## Technical Notes

- Next.js middleware (`middleware.ts`) uses `@clerk/nextjs` to protect all `/dashboard` routes. Unauthenticated requests are redirected to `/sign-in`.
- The Coordination API uses `@clerk/fastify` (or manual JWT verification via Clerk's JWKS endpoint) to validate every authenticated request.
- `auth()` is used in Server Components; `useAuth()` / `useOrganization()` in Client Components.
- The BFF pattern is enforced — the browser calls only Next.js API routes, which call the Coordination API using the Clerk session token. The internal Coordination API URL must not be exposed to the browser.
- SSE long-lived connections (Run Trace) use a separate token strategy — see Open Question #5 in `docs/requirements.md`.

---

## Dependencies

- Story 1.0 (Tenant Sign-up) — user and tenant rows must exist.
- Clerk project configured; `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` and `CLERK_SECRET_KEY` in environment.
- Next.js `middleware.ts` auth guard in place.
- Coordination API Fastify auth plugin configured.

---

## Out of Scope for This Story

- SSO/SAML sign-in (Enterprise tier, post-MVP).
- MFA enforcement at UI level (post-MVP).
- SSE token refresh strategy (tracked as Open Question #5 — resolved separately).

---

## Definition of Done

- [ ] Sign-in flow works end-to-end in staging.
- [ ] All `/dashboard` routes return 401 when accessed without a valid session.
- [ ] Org picker shown correctly for multi-org users.
- [ ] Sign-out clears session and redirects to landing page.
- [ ] Invalid/expired token on API call triggers frontend redirect to sign-in (no blank screen, no raw 401 shown to user).
- [ ] Unit tests: Fastify auth plugin rejects requests with missing/invalid JWT.
- [ ] Integration test: sign-in → dashboard → sign-out round trip.
