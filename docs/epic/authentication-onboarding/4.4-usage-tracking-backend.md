# Task 4.4 — Usage Tracking Backend (Backend)

**Story:** 4.0 — Trial Activation & Status Display  
**Skill:** backend-lead  
**Priority:** P0 — depends on Tasks 1.2, 2.3  

---

## What to do

Implement the `task_execution_usage` table, the usage-counting mechanism, and the `GET /usage/current-cycle` endpoint.

---

## Acceptance Criteria

**Prisma model**
- [ ] `task_execution_usage` model:
  - `id` (UUID)
  - `tenantId` (FK → tenants.id)
  - `workflowId` (String) — workflow identifier
  - `workflowName` (String)
  - `executedAt` (DateTime)
  - `billingCycleStart` (DateTime) — start of the current 30-day cycle (trial: signup date)

**Usage counting**
- [ ] A `recordTaskExecution(tenantId, workflowId, workflowName)` function inserts a row into `task_execution_usage`.
- [ ] This function is called by the task execution pipeline on job completion (stub the call site now; the actual pipeline job is in the Backend Services epic).

**API endpoint**
- [ ] `GET /usage/current-cycle` — requires auth. Returns:
  ```json
  {
    "tasksUsed": 12,
    "tasksLimit": 50,
    "cycleStart": "2025-05-01T00:00:00Z",
    "cycleEnd": "2025-05-31T00:00:00Z",
    "breakdown": [
      { "workflowName": "Jira Bug Fix", "count": 10 },
      { "workflowName": "Code Review", "count": 2 }
    ]
  }
  ```
- [ ] `tasksLimit` is read from the `PLAN_LIMITS` constant (Trial: 50 executions/cycle for MVP — confirm exact value).

**Testing**
- [ ] Unit test: `recordTaskExecution` inserts a row correctly.
- [ ] Unit test: `GET /usage/current-cycle` aggregates counts correctly.

---

## Notes

- For local dev, Redis-based counters (BullMQ) are not needed yet — read directly from `task_execution_usage` table.
- `billingCycleStart` for Trial plan = tenant's `createdAt` date. A 30-day rolling cycle is used (not calendar month).
