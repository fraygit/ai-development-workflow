# Task 6.1 — Trial Expiry Background Job (Backend)

**Story:** 6.0 — Trial Expiry & Upgrade Prompt  
**Skill:** backend-lead  
**Priority:** P0 — depends on Task 1.6  

---

## What to do

Implement the background jobs that set `plan_status = 'expired'` when a tenant's trial ends.

---

## Acceptance Criteria

**Delayed expiry job (BullMQ)**
- [ ] When a tenant is created (Task 1.6), a BullMQ delayed job `expire-trial:{tenantId}` is enqueued with delay = `trial_ends_at - now()` milliseconds.
- [ ] The job processor: sets `tenants.planStatus = 'expired'` where `id = tenantId AND planStatus = 'active'` (idempotent — uses a conditional update, not a blind write).
- [ ] Job is idempotent: running the same job twice does not corrupt the row (conditional `WHERE planStatus = 'active'` ensures no double-expiry).

**Daily sweep job (safety net)**
- [ ] A recurring BullMQ cron job runs daily at 00:05 UTC.
- [ ] The job finds all rows where `trial_ends_at < NOW() AND plan_status = 'active'` and sets `plan_status = 'expired'`.
- [ ] Catches any delayed jobs that were missed (e.g. worker was down).

**Local dev**
- [ ] BullMQ connects to the local Redis instance from Task 1.1 (`REDIS_URL=redis://localhost:6379`).
- [ ] Both jobs can be tested locally by creating a test tenant with `trial_ends_at = now() + 10 seconds`.

**Testing**
- [ ] Unit test: delayed job processor sets `planStatus = 'expired'` only when current status is `'active'`.
- [ ] Unit test: running the processor twice on an already-expired tenant changes nothing.
- [ ] Unit test: sweep job query selects only rows past expiry with `active` status.

---

## Notes

- BullMQ delayed jobs survive worker restarts if the job was already enqueued in Redis.
- For local dev, the cron job can be disabled by default and triggered manually via a test script.
