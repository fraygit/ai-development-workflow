# Task 3.7 — Plugin Credential Storage (Backend)

**Story:** 3.0 — Onboarding Plugin Setup Wizard  
**Skill:** backend-lead  
**Priority:** P0 — depends on Task 3.6  

---

## What to do

Implement the `POST /plugins/:pluginType` endpoint that stores a plugin's credentials after a passing connection test and creates/updates the `tenant_plugins` row.

---

## Acceptance Criteria

**Endpoint**
- [ ] `POST /plugins/:pluginType` — body: plugin-specific credential fields (same shape as the connection test body). Requires auth.
- [ ] The endpoint calls the connection test internally one more time before persisting — do not trust the frontend's claim that the test passed.
- [ ] On success: creates or updates `tenant_plugins` row with `enabled = true`; returns `201 { pluginType, enabled: true }`.
- [ ] On failed internal re-test: returns `422 { code: "CONNECTION_TEST_FAILED", message: "..." }`.

**Credential storage — local dev**
- [ ] For local development, credentials are written to a `.secrets/` directory (gitignored) as JSON files: `.secrets/tenant_{id}_{pluginType}.json`.
- [ ] `secretPath` on `tenant_plugins` stores `LOCAL:.secrets/tenant_{id}_{pluginType}.json`.
- [ ] A `SecretStore` interface with `get(path)` / `set(path, value)` is implemented with two adapters: `LocalSecretStore` (file-based) and `GcpSecretStore` (GCP Secret Manager — deferred, not implemented now). The active adapter is selected by `SECRET_STORE=local|gcp` env var.

**GCP Secret Manager (deferred)**
- [ ] GCP Secret Manager integration is NOT implemented in this task. The `GcpSecretStore` adapter stub returns `throw new Error("Not implemented")` until the GCP DevOps setup epic is completed.

---

## Notes

- `.secrets/` must be in `.gitignore`. Add it now even if it doesn't exist yet.
- The `SecretStore` interface pattern means the code is GCP-ready without requiring GCP setup for local dev.
- Path convention for future GCP: `tenants/{tenantId}/plugins/{pluginType}/{key}`.
