# Task 6.2 — Trial Expired Blocking Modal (Frontend)

**Story:** 6.0 — Trial Expiry & Upgrade Prompt  
**Skill:** frontend-lead  
**Priority:** P0 — depends on Tasks 4.1, 6.3  

---

## What to do

Implement the full-screen blocking modal shown to all users when their trial has expired. The modal prevents all dashboard interactions until the plan is upgraded.

---

## Acceptance Criteria

**Trigger**
- [ ] The dashboard layout Server Component checks `tenant.planStatus` on every render.
- [ ] If `planStatus = 'expired'`, the blocking modal is rendered server-side as part of the layout (not a client-side conditional).

**Modal content — Admin**
- [ ] Headline: "Your trial has ended"
- [ ] Body: "Your 14-day trial expired on [formatted date]. Upgrade to continue using AIDevFlow."
- [ ] Primary CTA: "Upgrade now" → navigates to `/dashboard/settings/billing`.
- [ ] Secondary CTA: "Contact support" → `mailto:support@aidevflow.io`.

**Modal content — Engineer / Viewer**
- [ ] Same headline and body.
- [ ] No upgrade CTA.
- [ ] Text: "Please ask your organisation admin to upgrade the account."
- [ ] Secondary CTA: "Contact support".

**Blocking behaviour**
- [ ] The modal cannot be dismissed — no close button, no `Escape` key, no click-outside-to-close.
- [ ] The underlying page content has `pointer-events: none` applied via Tailwind.
- [ ] Focus is trapped within the modal (keyboard users cannot Tab out to the blocked content).

**Accessibility**
- [ ] `role="dialog"`, `aria-modal="true"`, `aria-labelledby` pointing to the headline.
- [ ] Focus is set to the primary CTA on mount.

**Modal lifting**
- [ ] The dashboard layout refetches `planStatus` on every navigation (no cache on this field when `plan = 'trial'` and `planStatus = 'expired'`).
- [ ] Once `planStatus` returns to `'active'`, the modal is not rendered on the next route change.
- [ ] An SSE event (if implemented) can trigger an immediate lift without requiring navigation.

---

## Notes

- Build this as a dedicated `<TrialExpiredModal>` Server Component, not a re-use of shadcn/ui `Dialog` (since Dialog is typically dismissible).
- The blocking overlay and `pointer-events: none` should be on the layout wrapper div, not on the modal itself.
