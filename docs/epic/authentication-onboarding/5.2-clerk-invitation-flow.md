# Task 5.2 — Clerk Invitation Flow (Backend)

**Story:** 5.0 — User Invitation & Role Assignment  
**Skill:** backend-lead  
**Priority:** P1 — depends on Tasks 1.5, 2.3  

---

## What to do

Implement the invitation API endpoints that send Clerk organisation invitations and handle the resulting webhook to create `tenant_users` rows.

---

## Acceptance Criteria

**Send invitation**
- [ ] `POST /team/invite` — body: `[{ email, role }]`. Requires auth + Admin role.
- [ ] For each invite: calls Clerk's `createOrganizationInvitation` API with the `role` in the invitation metadata.
- [ ] Stores invitation in a `tenant_invitations` table (or uses Clerk's pending invitations list — decide which is source of truth).
- [ ] Idempotent: if `email` already has a pending invitation, revoke the old one and create a new one.
- [ ] Returns `201 { invitations: [{ email, role, status: 'pending' }] }`.

**Resend / revoke**
- [ ] `POST /team/invitations/:invitationId/resend` — revokes old Clerk invitation and creates a new one.
- [ ] `DELETE /team/invitations/:invitationId` — revokes the Clerk invitation.

**List**
- [ ] `GET /team` — returns both active members (from `tenant_users`) and pending invitations. Active members include `idpUserId`, `role`, `createdAt`. Pending invitations include `email`, `role`, `invitedAt`.

**Webhook handling (complement to Task 1.5)**
- [ ] `organizationMembership.created` from a non-creator invite: creates `tenant_users` row with `role` from invitation metadata.
- [ ] `organizationMembership.deleted`: sets `tenant_users.deletedAt = now()` and clears the Clerk session for that org (Clerk handles session invalidation automatically).

**Testing**
- [ ] Unit test: `POST /team/invite` calls Clerk API and creates invitation record.
- [ ] Unit test: duplicate invite replaces existing.

---

## Notes

- Clerk invitation email delivery is handled by Clerk — no need to implement email sending.
- The `role` in Clerk invitation metadata maps 1:1 to `tenant_users.role` — keep them in sync.
