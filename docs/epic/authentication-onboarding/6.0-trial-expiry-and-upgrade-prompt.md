# Story 6.0 — Trial Expiry & Upgrade Prompt

**Epic:** Authentication & Onboarding  
**Priority:** P0 — MVP critical path (required to enforce trial limits and drive conversion)  
**Roles involved:** Admin, Engineer, Viewer

---

## User Story

As a tenant whose 14-day trial has expired,  
I want to be shown a clear, blocking prompt to upgrade when I next log in,  
so that I understand my account is paused and I know exactly how to restore access.

---

## Acceptance Criteria

**Expiry detection**
- [ ] Trial expiry is determined server-side by comparing `tenants.trial_ends_at` to UTC now. The client never computes expiry.
- [ ] When the trial expires, `tenants.plan_status` is set to `'expired'`. This is done by a scheduled background job (daily cron or BullMQ delayed job triggered at `trial_ends_at`).
- [ ] The status change is idempotent — running the expiry job multiple times does not double-expire or corrupt the tenant row.

**Blocking modal on login (expired trial)**
- [ ] When a user from an expired-trial tenant logs in and lands on any `/dashboard` route, a full-screen blocking modal is displayed immediately.
- [ ] The modal cannot be dismissed — there is no close button or click-outside-to-close behaviour.
- [ ] Modal content (Admin):
  - Headline: "Your trial has ended"
  - Body: "Your 14-day trial expired on [date]. Upgrade to continue using AIDevFlow."
  - Primary CTA: "Upgrade now" — links to the billing upgrade flow.
  - Secondary CTA: "Contact support" — opens a mailto or support link.
- [ ] Modal content (Engineer / Viewer):
  - Same headline and body.
  - No upgrade CTA (they cannot upgrade).
  - Text: "Please ask your organisation admin to upgrade the account."
  - Secondary CTA: "Contact support".
- [ ] No other dashboard actions are available while the modal is shown — the underlying page is visually blocked (backdrop with `pointer-events: none` on the page content).

**Expiry enforcement on API**
- [ ] The Coordination API rejects all write operations (create, update, trigger, approve) with `403 TRIAL_EXPIRED` when `plan_status = 'expired'`.
- [ ] Read-only operations (GET run history, GET audit log) remain accessible — tenants can view past run data even after expiry.
- [ ] The 403 response body includes `code: "TRIAL_EXPIRED"` and a human-readable `message` so the frontend can differentiate from other 403s.

**Upgrade flow entry point**
- [ ] "Upgrade now" CTA navigates to the Billing page (`/dashboard/settings/billing`).
- [ ] The Billing page shows the available plans (Starter, Growth, Enterprise) and initiates Stripe Checkout when a plan is selected.
- [ ] The Stripe integration itself is out of scope for this story — the Billing & Trial epic owns it. This story only requires the CTA to navigate correctly and the blocking modal to lift once `plan_status` changes to `'active'`.

**Modal lifting on upgrade**
- [ ] After a successful upgrade (plan_status changes to `'active'`), the blocking modal disappears on the next dashboard load or SSE event (no manual refresh required).
- [ ] The trial banner (Story 4.0) is replaced with a plan indicator showing the new tier.

---

## Technical Notes

- Expiry job: a BullMQ delayed job is enqueued at tenant creation with a delay of `trial_ends_at - now()`. When it fires, it sets `plan_status = 'expired'` on the `tenants` row. A daily sweep job also catches any rows where `trial_ends_at < now() AND plan_status = 'active'` as a safety net.
- The dashboard layout Server Component (Next.js) checks `plan_status` on every render (same fetch used for trial banner in Story 4.0). If `plan_status = 'expired'`, it renders the blocking modal overlay as a Server Component — not a client-side check.
- `pointer-events: none` on the page content is enforced via Tailwind CSS class on the layout wrapper. The modal is rendered in a portal outside the main content tree to prevent scroll-through.
- API enforcement is in the Fastify route middleware — called after auth, before the route handler.
- The blocking modal is accessible: it is focusable, uses `role="dialog"`, `aria-modal="true"`, and traps focus within the modal until upgraded.

---

## Edge Cases

| Scenario | Expected behaviour |
|---|---|
| Trial expires while user is mid-session | Next navigation within the dashboard shows the blocking modal (middleware re-checks `plan_status` on route change). In-flight API calls return 403 with `TRIAL_EXPIRED`. |
| Trial expires during an active pipeline run | Existing running steps complete. New step dispatch calls from the runner return 403. The run is marked `'failed'` with reason `TRIAL_EXPIRED`. |
| Admin upgrades from one browser tab; another tab is open | Within 60s (SSE event or next cache revalidation), the blocking modal lifts in the other tab. |
| Multiple admins on the account | Any Admin can trigger the upgrade. All sessions see the modal lift once `plan_status` is updated. |

## Out of Scope for This Story

- Stripe Checkout and plan activation webhook handling (Billing & Trial epic).
- Trial extension (admin operation, no UI in MVP).
- Automated email notifications on expiry (post-MVP — email integration not in MVP scope).

## Definition of Done

- [ ] BullMQ expiry job fires at `trial_ends_at` and sets `plan_status = 'expired'`.
- [ ] Daily sweep job also catches any missed expirations.
- [ ] Blocking modal shown for all roles on next dashboard load after expiry; Admin sees upgrade CTA, others see "contact admin" message.
- [ ] Modal cannot be dismissed without upgrading.
- [ ] All write API calls return 403 `TRIAL_EXPIRED` when plan_status is `'expired'`.
- [ ] Read-only API calls (GET) remain accessible after expiry.
- [ ] Modal lifts automatically after plan_status returns to `'active'` — no manual refresh needed.
- [ ] Unit tests: expiry job idempotency.
- [ ] Unit tests: API middleware returns 403 on `TRIAL_EXPIRED` for write routes.
- [ ] Unit tests: read routes return 200 even when plan is expired.
- [ ] Integration test: sign-in with expired tenant → blocking modal → (mock upgrade) → modal lifts.
- [ ] Accessibility: modal uses `role="dialog"`, `aria-modal="true"`, focus is trapped.
