# Story 3.0 — Onboarding Plugin Setup Wizard

**Epic:** Authentication & Onboarding  
**Priority:** P0 — MVP critical path  
**Roles involved:** Admin

---

## User Story

As a newly signed-up admin,  
I want to be guided through a step-by-step plugin setup wizard immediately after creating my account,  
so that my organisation is fully configured before I attempt to create or run any workflow.

---

## Acceptance Criteria

**Wizard entry point**
- [ ] After successful sign-up (Story 1.0), the dashboard redirects to the onboarding wizard — not the dashboard home.
- [ ] The wizard is also accessible from the dashboard home via a dismissible "Complete your setup" banner until all required plugins are configured.
- [ ] Only Admin-role users see the wizard and setup banner. Engineers and Viewers are not prompted.

**Wizard steps (in order)**
1. **Source Control** — Connect GitHub App or GitLab Group Access Token.
2. **Issue Tracker** — Connect Jira (API token + base URL + project key + webhook secret).
3. **AI Provider** — Connect Anthropic (API key) or Azure AI Foundry endpoint.

- [ ] Steps are displayed as a numbered progress indicator (e.g. "Step 1 of 3").
- [ ] Each step shows only the required fields by default. Optional/advanced fields are behind an "Advanced" toggle.
- [ ] Each step includes a "Test connection" action that validates the credentials against the real external service before proceeding. The test result (success / error) is shown inline before the "Next" button becomes active.
- [ ] A step cannot be marked complete without a passing connection test.
- [ ] The wizard allows navigating back to a previous step to change credentials.
- [ ] If the admin closes the wizard mid-flow, progress is saved — reopening the wizard resumes from the first incomplete step.
- [ ] After all three steps pass, the wizard shows a "You're all set" completion screen with a CTA to "Create your first workflow" (links to Workflow Designer) or "Go to Dashboard".

**Error handling**
- [ ] If a connection test fails, an inline error is displayed explaining the failure in plain language (e.g. "Invalid Jira API token — check that the token has project-read permissions").
- [ ] No secret is persisted to GCP Secret Manager unless the connection test passes.
- [ ] If the Coordination API call fails (network error, 5xx), the user sees "Something went wrong. Please try again." with a retry button — not a raw error.

**Skip behaviour**
- [ ] The admin can skip individual steps (except Source Control, which is required to compile and deploy any workflow).
- [ ] Skipped steps appear as "Incomplete" in the progress indicator and in the Plugin Config section of the dashboard.

## Technical Notes

- Wizard state (current step, which steps are complete) is stored server-side in a `tenant_onboarding_state` field on the `tenants` row (or a dedicated `tenant_onboarding` table) so it survives page refresh.
- Client-side wizard step state uses Zustand (unsaved form drafts only — not persisted).
- Connection test calls are proxied through the Coordination API — credentials are never sent directly from the browser to Jira/GitHub/GitLab/Anthropic.
- On a passing connection test, the Coordination API stores the credential in GCP Secret Manager at path `tenants/{tenant_id}/plugins/{plugin_type}/{key}` and records the `tenant_plugins` row with `enabled = true`.
- The wizard UI is built with shadcn/ui components. Step indicator uses the `Steps` pattern from shadcn/ui (or a custom equivalent if not yet in the library).
- Accessibility: each step form must be keyboard-navigable. Focus moves to the first field of the new step on advance.

## Wizard Step Detail

### Step 1 — Source Control

| Field | Required | Notes |
|---|---|---|
| Provider | Yes | Radio: GitHub / GitLab |
| **GitHub:** GitHub App installation | Yes | OAuth redirect to GitHub App install flow; returns installation ID |
| **GitLab:** Group Access Token | Yes | Text input; scope: `api`, `read_repository`, `write_repository` |
| **GitLab:** Group namespace | Yes | Text input (e.g. `mycompany`) |

Connection test: verify the token/app can list repos in the specified namespace.

### Step 2 — Issue Tracker

| Field | Required | Notes |
|---|---|---|
| Provider | Yes | Radio: Jira (only option in MVP) |
| Jira base URL | Yes | e.g. `https://mycompany.atlassian.net` |
| Jira API token | Yes | Stored in Secret Manager |
| Jira user email | Yes | Used with the token for Basic Auth |
| Jira project key | Yes | e.g. `PROJ` |
| Webhook secret | No (Advanced) | Used to verify inbound Jira webhooks via HMAC |

Connection test: call `GET /rest/api/3/myself` on the Jira instance with the supplied credentials.

### Step 3 — AI Provider

| Field | Required | Notes |
|---|---|---|
| Provider | Yes | Radio: Anthropic / Azure AI Foundry |
| **Anthropic:** API key | Yes | Stored in Secret Manager |
| **Azure:** Endpoint URL | Yes | Azure AI Foundry resource endpoint |
| **Azure:** API key | Yes | Stored in Secret Manager |
| **Azure:** Deployment name | Yes | Model deployment name in Azure |

Connection test: make a minimal inference call (e.g. `{"role":"user","content":"ping"}`) and verify a non-error response.

## Dependencies

- Story 1.0 (tenant row created) and Story 2.0 (authenticated session).
- Plugin Config UI components (Plugin Configuration epic) — the wizard reuses the same plugin forms.
- Coordination API: `POST /plugins` endpoint and connection-test endpoints per plugin type.
- GCP Secret Manager access from Coordination API service account.

---

## Out of Scope for This Story

- Linear, GitHub Issues, SonarCloud plugins (post-MVP or Plugin Configuration epic).
- System user (Jira bot account) mapping — covered in Plugin Configuration epic.
- Runner label configuration — covered in Plugin Configuration epic.

---

## Definition of Done

- [ ] Wizard appears as first screen for new admin after sign-up.
- [ ] All three steps complete successfully in staging with real credentials.
- [ ] Connection tests fail gracefully with inline errors when invalid credentials are supplied.
- [ ] No credentials stored if connection test fails.
- [ ] Wizard state persists across page refresh.
- [ ] Completion screen shown after all steps pass.
- [ ] Keyboard navigation works across all wizard steps (WCAG 2.1 AA).
- [ ] Unit tests: connection test handlers per plugin type.
- [ ] Integration test: full wizard flow sign-up → step 1 → step 2 → step 3 → completion.
