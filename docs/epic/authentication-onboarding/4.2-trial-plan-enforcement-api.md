# Task 4.2 — Trial Plan Enforcement (Backend)

**Story:** 4.0 — Trial Activation & Status Display  
**Skill:** backend-lead  
**Priority:** P0 — depends on Task 2.3  

---

## What to do

Implement Fastify middleware that enforces Trial plan limits on the Coordination API, blocking operations that exceed the trial tier's constraints.

---

## Acceptance Criteria

**Middleware**
- [ ] A `checkPlanLimits` Fastify hook runs after auth (Task 2.3) on all write routes.
- [ ] Trial limits enforced:
  - Maximum 1 linked repository: `POST /repos` returns `403 { code: "PLAN_LIMIT_EXCEEDED", message: "Your trial plan allows 1 repository. Upgrade to link more." }` if the tenant already has a repo linked.
  - SSO endpoint (`POST /sso`): returns `403 { code: "PLAN_GATE", message: "SSO is not available on the Trial plan." }` if called.
- [ ] `planStatus = 'expired'` enforcement: all write routes return `403 { code: "TRIAL_EXPIRED" }` (full implementation in Story 6.0; add the check stub here so it's wired in).

**Response contract**
- [ ] 403 response body always includes `code` and `message` fields.
- [ ] The frontend can use `code` to show the correct inline error message.

**Testing**
> **TDD:** Write each test case below _before_ implementing the middleware. Run with `npx vitest run`. Confirm they fail (Red), implement to pass (Green), then refactor.
- [ ] Unit test: second `POST /repos` on a trial tenant → `403 PLAN_LIMIT_EXCEEDED`.
- [ ] Unit test: `POST /repos` on a tenant with 0 repos → `201` (allowed).
- [ ] Unit test: `POST /repos` on a non-trial plan → not blocked.

---

## Notes

- Enforcement is server-side only. Client-side UI messages are informational (Task 4.3 below) but are NOT the enforcement boundary.
- Limits are hardcoded per plan tier in a `PLAN_LIMITS` constant for MVP — no DB-stored plan config needed yet.
