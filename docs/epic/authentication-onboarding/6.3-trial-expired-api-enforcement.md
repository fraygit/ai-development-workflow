# Task 6.3 — Trial Expired API Enforcement (Backend)

**Story:** 6.0 — Trial Expiry & Upgrade Prompt  
**Skill:** backend-lead  
**Priority:** P0 — depends on Tasks 4.2, 6.1  

---

## What to do

Complete the `TRIAL_EXPIRED` enforcement in the Fastify plan-limits middleware (stubbed in Task 4.2) so that all write operations are blocked and read operations remain accessible after trial expiry.

---

## Acceptance Criteria

**Write route blocking**
- [ ] All non-GET routes return `403 { code: "TRIAL_EXPIRED", message: "Your trial has expired. Upgrade to continue using AIDevFlow." }` when `tenants.planStatus = 'expired'`.
- [ ] The check is in the `checkPlanLimits` middleware, after auth verification.

**Read route exceptions**
- [ ] `GET` routes are NOT blocked by the `TRIAL_EXPIRED` check — tenants can view run history, audit logs, and settings after expiry.
- [ ] Exception: `GET /usage/current-cycle` is also allowed (tenant should still see their usage).

**planStatus endpoint**
- [ ] `GET /tenant/status` — returns `{ planStatus, trialEndsAt, plan }`. Used by the frontend dashboard layout to determine whether to show the blocking modal. No caching — always reads live from DB.

**Testing**
- [ ] Unit test: `POST /repos` with `planStatus = 'expired'` → `403 TRIAL_EXPIRED`.
- [ ] Unit test: `GET /runs` with `planStatus = 'expired'` → `200` (not blocked).
- [ ] Unit test: `GET /tenant/status` returns current `planStatus` accurately.
- [ ] Integration test: expire a test tenant via the job (Task 6.1) → confirm write routes return `403`, read routes return `200`.

---

## Notes

- The distinction is HTTP method: `GET` = allowed, everything else = blocked when expired. Apply this as a simple method check in the middleware — no need to enumerate allowed routes.
- `GET /tenant/status` is the polling endpoint for the frontend layout. Keep it lightweight: single DB read, no joins.
