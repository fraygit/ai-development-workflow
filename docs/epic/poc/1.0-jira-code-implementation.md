# Story 1.0 — Jira → Implement → PR

**Epic:** POC  
**Tasks:** 1.1, 1.2, 1.3  
**Priority:** P0

---

## Goal

A Jira ticket assigned to a configured system user (bot account) triggers a GitHub Actions workflow. Claude Code CLI reads the ticket, implements the requested change in the sample Node.js project, commits to a feature branch, and opens a PR targeting `main`.

The full loop runs without a human touching the keyboard after the Jira assignment.

---

## Acceptance Criteria

- [ ] A Jira ticket with a clear, self-contained task (see sample below) triggers the workflow when assigned to the system user, either via:
  - (a) `workflow_dispatch` in the GitHub UI (for local testing without the bridge), or
  - (b) The Jira webhook bridge (Task 1.3) receiving the Jira assignment event.
- [ ] Claude Code creates branch `feature/<ticket-id>-ai-implementation`, implements the change, and pushes it.
- [ ] A PR is opened targeting `main` with the ticket ID in the title and a link to the Jira ticket in the body.
- [ ] The PR description includes a warning that auto-merge is disabled and a human must review.
- [ ] No secrets (`ANTHROPIC_API_KEY`, `JIRA_API_TOKEN`) appear in the PR body, commit message, or diff.
- [ ] The workflow completes within 5 minutes from trigger to PR created.
- [ ] The workflow fails gracefully (non-zero exit) if the Jira ticket cannot be fetched, and the GitHub Actions run is marked as failed.

---

## Sample POC Ticket

Use a ticket with a small, unambiguous implementation task so the POC is easy to verify:

> **Summary:** Add a `greet(name)` function to `src/index.js`  
> **Description:** Add and export a `greet(name)` function that returns the string `Hello, {name}!`. Add a test for it in `test/index.test.js`. Follow the existing code style.

This is intentionally trivial — the goal is to prove the automation loop works, not the AI's coding capability.

---

## Notes

- The sample Node.js project in `src/` and `test/` should be minimal but real: a `package.json` with a test runner (Vitest or Node's built-in test runner), one source file, and one test file. This gives Claude enough context to follow conventions without overwhelming the POC.
- Story 1 does NOT need to handle the case where Claude's implementation fails tests. That is a post-POC concern (retry logic, quality gate). For the POC, if tests fail it is visible in the PR CI — that is sufficient.
