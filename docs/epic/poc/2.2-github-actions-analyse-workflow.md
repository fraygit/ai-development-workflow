## What to do

Write `.github/workflows/jira-analyse.yml` in the POC repo. This workflow is intentionally simpler than `jira-implement.yml` — no git operations, no branch, no PR. It just fetches the Jira ticket + comments, runs Claude with the `ticket-analyser` skill, and exits.

---

## File to Create

**Path in POC repo:** `.github/workflows/jira-analyse.yml`

```yaml
name: Jira — Analyse Ticket

on:
  repository_dispatch:
    types: [jira-ticket-assigned-for-analysis]
  workflow_dispatch:
    inputs:
      jira_ticket_id:
        description: 'Jira ticket ID (e.g. POC-2)'
        required: true
        type: string
      return_assignee_account_id:
        description: 'Jira accountId to reassign ticket to after analysis'
        required: true
        type: string

jobs:
  analyse:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Resolve inputs
        id: inputs
        run: |
          if [ "${{ github.event_name }}" = "repository_dispatch" ]; then
            echo "ticket_id=${{ github.event.client_payload.ticket_id }}" >> $GITHUB_OUTPUT
            echo "return_assignee=${{ github.event.client_payload.return_assignee_account_id }}" >> $GITHUB_OUTPUT
          else
            echo "ticket_id=${{ inputs.jira_ticket_id }}" >> $GITHUB_OUTPUT
            echo "return_assignee=${{ inputs.return_assignee_account_id }}" >> $GITHUB_OUTPUT
          fi

      - name: Fetch Jira ticket and comments
        id: jira
        env:
          JIRA_BASE_URL: ${{ secrets.JIRA_BASE_URL }}
          JIRA_API_TOKEN: ${{ secrets.JIRA_API_TOKEN }}
          JIRA_USER_EMAIL: ${{ secrets.JIRA_USER_EMAIL }}
        run: |
          TICKET_ID="${{ steps.inputs.outputs.ticket_id }}"
          AUTH=$(echo -n "$JIRA_USER_EMAIL:$JIRA_API_TOKEN" | base64 -w0)

          ISSUE=$(curl -sf \
            -H "Authorization: Basic $AUTH" \
            -H "Content-Type: application/json" \
            "$JIRA_BASE_URL/rest/api/3/issue/$TICKET_ID")

          COMMENTS_RAW=$(curl -sf \
            -H "Authorization: Basic $AUTH" \
            -H "Content-Type: application/json" \
            "$JIRA_BASE_URL/rest/api/3/issue/$TICKET_ID/comment")

          SUMMARY=$(echo "$ISSUE" | jq -r '.fields.summary')

          DESCRIPTION=$(echo "$ISSUE" | jq -r '
            [.fields.description.content[]?.content[]?
              | select(.type == "text") | .text] | join(" ")' 2>/dev/null || echo "")

          # Extract plain text from all comment bodies
          COMMENTS=$(echo "$COMMENTS_RAW" | jq -r '
            [.comments[]
              | .body.content[]?.content[]?
              | select(.type == "text") | .text] | join(" | ")' 2>/dev/null || echo "")

          echo "summary=$SUMMARY" >> $GITHUB_OUTPUT
          {
            echo "description<<EOF"
            echo "$DESCRIPTION"
            echo "EOF"
          } >> $GITHUB_OUTPUT
          {
            echo "comments<<EOF"
            echo "$COMMENTS"
            echo "EOF"
          } >> $GITHUB_OUTPUT

      - name: Install Claude Code CLI
        run: npm install -g @anthropic-ai/claude-code

      - name: Run ticket-analyser skill
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
          JIRA_TICKET_ID: ${{ steps.inputs.outputs.ticket_id }}
          JIRA_SUMMARY: ${{ steps.jira.outputs.summary }}
          JIRA_DESCRIPTION: ${{ steps.jira.outputs.description }}
          JIRA_COMMENTS: ${{ steps.jira.outputs.comments }}
          JIRA_BASE_URL: ${{ secrets.JIRA_BASE_URL }}
          JIRA_API_TOKEN: ${{ secrets.JIRA_API_TOKEN }}
          JIRA_USER_EMAIL: ${{ secrets.JIRA_USER_EMAIL }}
          RETURN_ASSIGNEE_ACCOUNT_ID: ${{ steps.inputs.outputs.return_assignee }}
        run: |
          claude --no-interactive \
            --allowedTools "Bash" \
            -p "$(cat .claude/skills/ticket-analyser/SKILL.md)"
```

## Acceptance Criteria

- [ ] Workflow file exists at `.github/workflows/jira-analyse.yml` in the POC repo.
- [ ] `workflow_dispatch` trigger accepts both `jira_ticket_id` and `return_assignee_account_id`.
- [ ] `repository_dispatch` listens for `jira-ticket-assigned-for-analysis` and reads both fields from `client_payload`.
- [ ] Both Jira issue details AND comments are fetched before Claude runs.
- [ ] Claude is invoked with `--allowedTools "Bash"` only — no filesystem tool access.
- [ ] No `contents: write` or `pull-requests: write` permissions needed (simpler permissions than `jira-implement.yml`).

**Manual test sequence:**
- [ ] Go to GitHub → Actions → `Jira — Analyse Ticket` → Run workflow → enter a ticket ID and a Jira accountId for `return_assignee_account_id`.
- [ ] Workflow completes green.
- [ ] Open the Jira ticket — a comment with the structured plan has been posted.
- [ ] Ticket is now assigned to the accountId you entered.

---

## Notes

- `--allowedTools "Bash"` is intentional. The ticket-analyser skill only uses `curl` via Bash to call Jira. Restricting tools prevents Claude from accidentally touching the filesystem in this read-only analysis workflow.
- The `return_assignee_account_id` for `workflow_dispatch` is required input. For `repository_dispatch`, the bridge passes it in `client_payload`. For the POC, you can also set it as a repo-level secret (`JIRA_RETURN_ASSIGNEE_ACCOUNT_ID`) and reference it directly — whichever is simpler.
- No `npm ci` step needed here since the ticket-analyser skill does not interact with the Node.js project source code.
