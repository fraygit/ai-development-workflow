# Story 2.0 — Jira → Analyse → Comment

**Epic:** POC  
**Tasks:** 2.1, 2.2  
**Priority:** P1 — tackle after Story 1.0 is proved working

---

## Why this second workflow?

Story 1 (implement → PR) is the flagship but it's also the most complex — it involves git operations, branch creation, and Claude editing files. Story 2 is deliberately simpler and tests a different, equally important capability: **pure AI analysis with no side effects on the codebase**.

This maps directly to the `ticket-analyst` built-in task in the full AIDevFlow platform.

---

## Goal

A Jira ticket assigned to a second system user triggers a GitHub Actions workflow. Claude reads the full Jira thread (description + all comments), writes a structured implementation plan, posts it back to Jira as a comment, and reassigns the ticket to the human tech lead.

No git operations. No PR. Just read → think → post.

---

---

## Acceptance Criteria

- [ ] A Jira ticket assigned to the **analyst system user** (different accountId from the implementer bot) triggers `jira-analyse.yml`.
- [ ] Claude reads `$JIRA_SUMMARY`, `$JIRA_DESCRIPTION`, and `$JIRA_COMMENTS` and produces a plan in the exact structure defined in the skill.
- [ ] The plan is posted as a Jira comment on the original ticket (visible in Jira within 60 seconds of the trigger).
- [ ] The ticket is reassigned to `$RETURN_ASSIGNEE_ACCOUNT_ID` after the comment is posted.
- [ ] The workflow completes in under 3 minutes.
- [ ] If Jira ticket description is empty or the ticket cannot be fetched, the workflow fails clearly — no comment is posted.

---

## Sample POC Ticket for Story 2

> **Summary:** Plan: add input validation to the `greet` function  
> **Description:** The `greet(name)` function (added in POC-1) currently does not validate its input. Plan how to add input validation — what should be checked, what error should be thrown, which tests need to be added.

This gives Claude enough context to write a useful plan without needing filesystem access.
